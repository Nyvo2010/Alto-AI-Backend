# TOOL-INSTRUCTIES — Naming conventions voor settings keys

Doel: geef een eenduidige conventie zodat instellingen die in JSON worden opgeslagen dezelfde, voorspelbare keys hebben.

Belangrijkste regels
- Globale vorm: `<integration>.<setting_name>`
  - `<integration>` is een korte, lower-case identificator (prefix) voor de integratie, bijvoorbeeld:
    - Discord → `dc` of `discord` (we gebruiken `dc` in voorbeelden hierboven, maar kies één en houd die aan projectbreed)
    - Trello → `trello`
    - GitHub → `github`
    - Slack → `slack`
    - Notion → `notion`
    - Filesystem → `fs`
  - `<setting_name>`: lower_snake_case (a.k.a. lower case with underscores).
- Gebruik punten (.) om namespaces te maken (integration scope + setting path).
- Types: bewaar type-informatie niet in de key. Types worden beheerd in schema/validation (bv. JSON Schema).
- Geen spaties, geen speciale tekens behalve underscore (_) en dot (.).
- Versie/deprecatie: indien nodig, gebruik suffix `@v2` niet in key zelf — gebruik metadata of migratie-tabellen. Keys moeten stabiel blijven.
- Wanneer een setting een object of map is, specificeer de key als prefix en gebruik nested JSON:
  - Voor `trello.lists_by_board`:
    ```json
    {
      "trello": {
        "lists_by_board": {
          "board_id_123": ["list_id_a", "list_id_b"]
        }
      }
    }
    ```
- Gebruik booleans (`true`/`false`) voor on/off flags; integers voor times/limits; arrays for lists.

Conventies en voorbeelden
- Prefixes (consistente korte identifiers)
  - Discord: `dc` (example: `dc.allowed_channels`)
  - Trello: `trello` (example: `trello.allow_create`)
  - Filesystem: `fs` (example: `fs.write_paths`)
  - Web/search: `web` or `search` (example: `web.blacklist`)
  - Images: `img` (example: `img.save_output`)
- Key examples:
  - `dc.message_cooldown_seconds` → integer seconds
  - `trello.default_labels` → array of label names or objects
  - `web.summarize_results` → boolean
  - `notion.allowed_pages` → array of page IDs

Schema / Validation
- Hanteer JSON Schema of Pydantic/Marshmallow/etc in backend om:
  - Type checks
  - Required/optional flags
  - Default values
  - Allowed values/ranges
- Sla instellingen op in eenduidig JSON-object per integratie of in één centrale settings-object:
  - Centrale vorm (voorbeeld):
    ```json
    {
      "dc": {
        "allowed_servers": ["123", "456"],
        "message_cooldown_seconds": 5,
        "moderation_filter_enabled": true
      },
      "trello": {
        "boards": ["boardA", "boardB"]
      }
    }
    ```

Naming decisions to make (choose one project-wide and document it)
- Short prefix vs full name (e.g., `dc` vs `discord`) — het is aan te raden één keuze te maken voor het hele project.
- Gebruik underscore-separated words (`lower_snake_case`) voor setting names.
- Gebruik dot-separated namespace for integration scoping.

Migration guidance
- Als keys moeten veranderen:
  - Voeg nieuwe key naast oude key met de nieuwe naam en markeer oude key als deprecated in metadata.
  - Voeg een migratie-script die oude keys naar nieuwe keys kopieert.
  - Houd backward compatibility voor minimaal één release.

Praktische tips voor implementatie
- Maak een centraal bestand (YAML/JSON) met de default settings + keys (kan automatisch dienen als single source of truth).
- Gebruik die bron om frontend en backend te genereren of te valideren (documentatie, forms, UI labels).
- Documenteer human-readable labels en descriptions naast keys (voor UI).
- Schrijf tests die checken of alle keys voldoen aan conventie (regex: `^[a-z][a-z0-9_]*(\.[a-z0-9_]+)*$`).

Kort voorbeeld van key-to-UI mapping
- key: `dc.allowed_channels`
- label: "Toegestane channels"
- type: array of strings (channel IDs)
- description: "Lijst met Discord channel IDs waar de bot mag posten"
